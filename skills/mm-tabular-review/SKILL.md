# M&A Contract Review — General Key Terms (Parent Skill)

This is the **parent skill**. It owns document intake, batching, the General Key Terms tabular
review (Sheet 1), and the Summary sheet (Sheet 3). It does not itself contain the Change of
Control / Transfer field definitions — those live in the **child skill**,
`mm-coc-tabular-review`. When the user wants that deep-dive, this skill hands off to the child
skill at the CoC Gate (Phase 3.5) and the child skill writes Sheet 2 into the same workbook. See
"Handing off to the child skill" in Phase 3.5 below.

---

## What this skill produces

A single Excel workbook with two or three sheets depending on two independent choices the user
makes before the workbook is built: which **key terms** to use for Sheet 1 (Phase 0), and whether
to run the **Change of Control deep-dive** (Phase 3.5).

**Sheet 1 — Tabular Review.** One row per document (or document package), covering either the 20
standard M&A contract diligence data points built into this skill, or the user's own custom key
terms, or a combination of both — whatever the user selected at the Key Terms Selection Gate
(Phase 0). Every data row is immediately followed by a source-citation row that records which
section of the document each answer comes from. Always produced.

**Sheet 2 — Change of Control / Transfer Key-Terms Analysis.** Produced only if the user confirms
at the CoC Gate (Phase 3.5). This sheet is entirely owned and generated by the child skill,
`mm-coc-tabular-review`, which has its own Field Selection Gate for choosing default vs. custom
CoC fields. See Phase 3.5 for the handoff.

**Sheet 3 — Summary.** Deal-team action items and a verification reminder. Always produced, but
its content adjusts based on whether Sheet 2 was produced. The Summary does NOT include a
state-count table — that information is implicit in the colour coding of Sheet 1 and would be
redundant noise in the Summary.

## Phase 0 — Key Terms Selection Gate: Ask before starting any analysis

Before reading any document or extracting any field, ask the user which key terms Sheet 1 should
use. This question does not depend on the uploaded documents, so ask it as soon as the user
invokes this skill (it can be asked in the same turn as acknowledging the upload).

Post the following message in chat:

---

> **Before I start the review** — Sheet 1 normally covers 20 standard general key terms (parties,
> term, governing law, confidentiality, assignment, change of control, termination, liability,
> indemnification, insurance, IP, dispute resolution, survival, and more).
>
> Would you like me to:
> 1. **Use the default 20 key terms** — the standard set built into this skill.
> 2. **Add my own custom key terms to the defaults** — keep all 20 defaults and append additional
>    columns you define.
> 3. **Replace the defaults with my own custom key terms** — skip the built-in 20 and use only the
>    key terms you specify.
>
> Reply with 1, 2, or 3. If you choose 2 or 3, tell me the key terms you want — for each one, a
> short name and what it should capture (e.g., "Renewal Options — every renewal/extension right,
> notice deadline, and any conditions, stated verbatim").

---

Wait for the user's reply before proceeding to Phase 1 (Intake). Do not guess at custom key terms
the user has not stated — if the user picks option 2 or 3 but gives an incomplete or vague list,
ask a follow-up question naming what is missing (e.g., "You gave me a name but not what it should
capture — what should the '<name>' column look for?") rather than inventing retrieval
instructions on their behalf.

| User reply | Action |
|-----------|--------|
| 1 (or any clear preference for the defaults) | Use the 20 default columns defined in Phase 2A, unchanged. |
| 2, with a complete list of custom terms | Use all 20 default columns from Phase 2A, then append the user's custom terms as additional columns, numbered starting at 21. Build each custom column's extraction rule per "Building custom column definitions" below. |
| 3, with a complete list of custom terms | Discard the 20 default columns from Phase 2A entirely. Use only the user's custom terms as the Sheet 1 columns, numbered starting at 1. Build each custom column's extraction rule per "Building custom column definitions" below. |
| Ambiguous or incomplete | Ask a clarifying follow-up. Do not proceed to Phase 1 until the key terms for Sheet 1 are fully settled. |

This choice applies to the entire review (all batches). Do not re-ask it for later batches. If the
user changes their mind mid-project, apply the new set of key terms from that point forward and
note the switch in the Sheet 3 Run Information, and re-confirm which already-processed documents
(if any) need to be revisited for the newly added/changed columns.

### Building custom column definitions

For every custom key term the user supplies, construct a column definition with the same
structure and rigour as the default Sheet 1 columns in Phase 2A:

- **Column name:** the short name the user gave it.
- **What to extract:** the user's description of what the column should capture, tightened into a
  clear retrieval instruction (what to look for, what to include, what to exclude).
- **Discipline:** custom Sheet 1 columns follow the exact same extraction discipline as the default
  columns — verbatim or close-paraphrase from the document's own text, the same four cell states
  (Answered / Not present / Unclear / Needs review), the same source-citation row rules, and the
  same absolute prohibitions (no market-standard comparisons, no advisory language, no
  speculation) described in Phase 2 below. Do not relax these rules for custom columns.
- **Length:** unless the user specifies otherwise, match the length and style of the closest
  comparable default column (a short verbatim clause extract, not a paragraph of analysis).

If the user's description of a custom term is itself advisory or evaluative (e.g., "flag whether
this deal is risky"), ask them to reframe it as a factual extraction target, since Sheet 1 is a
factual record — evaluative judgment belongs in Sheet 3, not Sheet 1.

## Phase 1 — Intake: Read and organise what was uploaded

### 1.1 — Identify what has been uploaded

Check what the user has provided. There are three possible scenarios:

**Scenario A — Individual files.** The user has uploaded one or more individual contract files
(e.g., three separate .docx files). Treat each file as one document and give it one row in the
review.

**Scenario B — A folder or document package.** The user has uploaded a folder or zip that contains
a base agreement plus related files such as amendments, addenda, schedules, exhibits, statements
of work, side letters, or guaranties. In this case, treat the entire package as one document (one
row in the review) but read every file in the package before extracting any field. The base
agreement and all its amendments and schedules are one integrated contract — a term in Amendment 3
overrides the same term in the base agreement, and a schedule defines obligations that are part of
the deal. Never review the base agreement in isolation when amendments exist.

**Scenario C — Mixed.** The user has uploaded a combination of individual files and packages.
Apply Scenario A logic to standalone files and Scenario B logic to packages.

When identifying packages, look for naming patterns such as:
- "Agreement + Amendment 1 + Amendment 2 + Schedule A"
- "MSA + SOW_01 + SOW_02 + Exhibit_Data_Protection"
- "Lease + Lease_Amend_2019 + Lease_Amend_2022 + Schedule_Legal_Description"
- A folder whose name matches the base agreement

If the relationship between uploaded files is not clear, ask the user before proceeding:
> "I can see [list of files]. Should I treat these as separate contracts (one row each) or as a
> single contract package (one row for the whole set)?"

### 1.2 — Handle password-protected documents

Before reading any file, attempt to open it. If a file is password-protected or encrypted and
cannot be opened, do not skip it silently. Instead:

- Tell the user clearly: "The file [filename] appears to be password-protected and I cannot read
  its contents."
- Ask: "Please provide the password for this file, or if you are unable to share the password,
  let me know and I will mark this document as `needs_review` with a note explaining it could not
  be read."
- Wait for the user's response before proceeding with that document.
- If the user provides a password, attempt to open the file again using it.
- If the user confirms they cannot provide a password, give that document a full row where every
  cell is marked `needs_review` and the notes column explains: "Document could not be read —
  password-protected. Manual review required."
- Do not assume a file is readable just because it was uploaded. Always confirm it opened
  successfully before extracting fields.

### 1.3 — Read every document in full

Read the complete text of every file before extracting a single field. Do not skim, do not chunk,
do not rely on memory of what such contracts "usually" say. The specific language in front of you
is what matters.

For a document package, read the base agreement first, then read each amendment and schedule in
chronological or logical order. As you read, note where later documents modify, replace, or add to
earlier provisions. When you extract a field, your answer must reflect the final, integrated
position — not the base agreement alone.

If a file cannot be read for any reason other than password protection (corrupted file,
unsupported format, empty file), tell the user and mark that document `needs_review` with a note
explaining the reason.

---

### 1.4 — Batching (processing in groups of five)

Handle documents and document packages in small batches rather than all at once. This keeps
accuracy high, reduces errors on long uploads, and lets results reach the user in manageable
stages.

- Process a **maximum of 5 documents or document packages per batch**. A package (base agreement
  plus its amendments, schedules, exhibits, side letters, and related files, per Scenario B) counts
  as one unit toward that limit — not each file inside it.
- Sort and process documents/packages in name or number order (smallest to largest) unless the
  user instructs otherwise.
- Complete Sheet 1 for the first 5 documents/packages, and Sheet 2 as well if the CoC gate has
  already been confirmed YES (see the batching/CoC-gate interaction rule in Phase 3.5), then
  immediately add those rows to the Excel workbook.
- After the first batch is written, continue with the next batch (documents/packages 6–10), adding
  the new rows to the **same** workbook without overwriting the previous ones. Repeat for all
  remaining batches (11–15, 16–20, and so on) until every uploaded document/package has been
  processed, always appending to the same consolidated workbook.
- Maintain a single workbook throughout — do not create a separate file per batch.
- After each batch, save the updated workbook containing every document/package processed so far,
  and include a running note in the work-product banner (Row 1 of Sheet 1 and Sheet 2) or in the
  Sheet 3 Run Information: `"Batch X of Y — Processed: [list of document/package names]. Remaining
  documents will be processed in subsequent batches and added to this same workbook."`
- After completing each batch, report to the user in chat: which documents/packages were just
  processed, which remain, and any issues found during that batch (password-protected files,
  unreadable files, ambiguous groupings).
- Do not wait for user confirmation before starting the next batch — continue processing
  automatically, batch after batch, until everything uploaded has been covered. Only pause between
  batches if the user has explicitly asked to review each batch before continuing.
- If a single batch contains more than 5 documents/packages worth of upload (e.g., the user drops
  20 files at once), split it into multiple batches per the rule above rather than processing them
  all in one pass.

## Phase 2 — Extract fields

> Everything in Phase 2 and Phase 2A runs once per batch, on the up-to-5 documents/packages in
> that batch, before moving to the next batch (see Phase 1.4).


### The four cell states

Every cell in Sheet 1 and Sheet 2 must carry one of four states. Never leave a cell blank — a
blank cell hides information that the deal team needs.

| State | Background | When to use |
|-------|-----------|-------------|
| **Answered** | White | You found substantive content and extracted it fully. Report what the clause says using the document's own terms. Note express gaps where silence is visible from the text itself. |
| **Not present** | Gray | You read the entire document and confirmed this topic is not addressed anywhere. Use only when certain — if any doubt exists, use Unclear. |
| **Unclear** | Yellow | The document addresses the topic but the drafting is ambiguous, contradictory, or incomplete. Describe what is written and why it is unclear. Do not interpret or resolve the ambiguity. |
| **Needs review** | Yellow | You found content but a human must make the call — judgment call, unusual structure, or apparent drafting error. State what the document says and flag what requires human judgment. |

> **Not present** and **Unclear** are different signals. A document silent on limitation of
> liability (not present) differs from a document with a liability clause whose scope is ambiguous
> (unclear). Use them precisely.

### The source-citation rule

After every data row, write a source-citation row. In each column of the citation row, write only
the section number(s) and heading(s) that support the answer directly above — for example:
`§14 Assignment; §4.2 Transfer by Tenant`. If the answer is not present, write `N/A — not present`.

**Citation format rules — strictly enforced:**
- Write only the section reference and its heading. Nothing else.
- Do NOT prefix citations with "Col X –", column numbers, or column names.
- Do NOT add explanatory notes, parenthetical comments, or context in the citation row — those
  belong in the data row.
- ✅ Correct: `§3.1 Definition of Confidential Information`
- ❌ Incorrect: `Col 6 – §3.1 Definition of Confidential Information`

Never invent a citation. If you cannot locate the source section, set the cell to `needs_review`
and write "Source location unavailable" in the citation row.

For document packages, note which file the citation comes from when the relevant text is in an
amendment or schedule rather than the base agreement — for example:
`Amendment 2, §3 — supersedes Base Agreement §7.1`

### Extraction discipline for all Sheet 1 columns — MANDATORY

**Every answer in Sheet 1 must consist exclusively of facts drawn verbatim or in close paraphrase
from the document's own text.** This is the single most important rule for Sheet 1 extraction.
Apply it without exception.

**What this means in practice:**

Report what the clause says by using the document's own words, terms, defined phrases, and
sentence structure. Where the document uses a specific defined term (e.g., "Receiving Party",
"Transfer", "Confidential Information"), use that same term in your answer. Where the document
provides a numbered list of obligations or exclusions, replicate that structure.

Identify express gaps by stating what the document does not address, but only where the absence is
apparent from the text itself — for example, "the document does not define the notice period" or
"no standard of care is stated". Do not add observations about what is missing unless you can
point to the specific place in the document where you looked and found nothing.

**Absolute prohibitions for Sheet 1 cells — no exceptions:**

- ❌ Do NOT compare to "market standard", "typical agreements", or any external benchmark.
- ❌ Do NOT speculate on how a court, arbitrator, or regulator might interpret the clause.
- ❌ Do NOT write advisory phrases such as "the acquirer should", "this is a material risk",
  "in practice this means", "this is unusual", "this is favourable / unfavourable".
- ❌ Do NOT infer intent. If the text is unclear, say so and describe the two possible readings
  using the document's own language.
- ❌ Do NOT add context about what provisions of this type "usually" contain.
- ❌ Do NOT make recommendations. Sheet 1 is a factual record.

The only permitted forms of editorial judgment in Sheet 1 are:
1. The choice of cell state (answered / not present / unclear / needs review)
2. The factual identification of express textual gaps
3. In Column 20 (Flags & Notes), the identification of drafting anomalies, apparent errors, and
   express gaps — stated as factual observations with section references, not as legal opinions

**Answer length rule for Sheet 1 — strictly enforced:**
Every Sheet 1 cell answer must match the length and style of the column examples below — short,
verbatim or close-paraphrase extracts, not paragraphs of analysis. If the example for a column
is two sentences, your answer should be roughly two sentences. If the example is a short numbered
list, your answer should be a short numbered list. Do not write more than the example shows.
The analytical, explanatory voice used in Sheet 2 (Phase 4) must never appear in any Sheet 1 cell.
Sheet 1 answers are extractions. Sheet 2 answers are analysis. They are different tasks.

## Phase 2A — Column definitions (Sheet 1)

> **These are the default 20 columns.** They are used exactly as written if the user chose option 1
> at the Key Terms Selection Gate (Phase 0). If the user chose option 2, use all of these plus the
> custom columns appended after Column 20. If the user chose option 3, skip this section entirely
> and use only the user's custom columns instead, applying the same discipline described in Phase 2
> and the "Building custom column definitions" guidance in Phase 0.

### Column 1 — Document
Exact descriptive name of the contract.

*Example:* `Commercial Net Lease (base agreement + First Amendment dated June 15, 2024 + Schedule A – Rules and Regulations)`

---

### Column 2 — Doc Type
The type of agreement as stated in the document.

*Example:* `Commercial Net Lease`

---

### Column 3 — Parties
Full legal names of all parties, their jurisdiction of incorporation/organization as stated, and
their roles. Note any defined collective parties verbatim.

*Example:*
```
"Landlord" means ABC Realty Partners LLC, a Delaware limited liability company.
"Tenant" means TechFlow Solutions Inc., a California corporation.
```

---

### Column 4 — Effective Date / Term
Verbatim: commencement/effective date, expiry date or duration, auto-renewal language, extension
options, and any perpetual/indefinite term language. Note changes from amendments.

*Example:*
```
"The term of this Lease shall commence on the Commencement Date of January 1, 2023 and shall
expire on December 31, 2027."
"This Lease shall automatically renew for one (1) additional period of five (5) years unless
either party gives written notice of non-renewal at least one hundred eighty (180) days prior
to expiration."
```

---

### Column 5 — Governing Law and Jurisdiction
Verbatim: chosen governing law, courts or arbitration forum, exclusivity of jurisdiction, seat of
arbitration, and any prevailing party fees language.

*Example:*
```
"This Lease shall be governed by the laws of the State of California. The parties submit to the
exclusive jurisdiction of the state and federal courts located in San Francisco County, California.
The prevailing party in any action shall be entitled to recover reasonable attorneys' fees."
```

---

### Column 6 — Confidential Information Definition
Full verbatim definition of "Confidential Information", including all listed categories. State
whether information must be marked "Confidential".

*Example:*
```
"Confidential Information means any information disclosed by one party to the other, whether
orally or in writing, that is designated as confidential or that reasonably should be understood
to be confidential given the nature of the information and the circumstances of disclosure,
including but not limited to financial data, business plans, customer lists, and proprietary
technology."
```

---

### Column 7 — Exclusions from Confidential Information
Every carve-out/exclusion exactly as written, including any conditions or language about
combinations of information.

*Example:*
```
"Confidential Information shall not include information that: (i) is or becomes generally
available to the public without breach of this Agreement; (ii) was known to the Receiving Party
prior to disclosure; (iii) is received from a third party without breach of any obligation owed
to the Disclosing Party; or (iv) is independently developed by the Receiving Party without use
of the Disclosing Party's Confidential Information."
```

---

### Column 8 — Non-Disclosure and Non-Use Obligations
Core obligations of the receiving party verbatim. List permitted disclosees and the stated
standard of care.

*Example:*
```
"The Receiving Party shall not disclose or use the Disclosing Party's Confidential Information
except as necessary to perform its obligations or exercise its rights under this Agreement.
The Receiving Party shall protect the Confidential Information using the same degree of care as
it uses to protect its own confidential information of similar nature, but in no event less than
reasonable care."
```

---

### Column 9 — Compelled Disclosure
Exact procedure and notice requirements when disclosure is legally compelled.

*Example:*
```
"If the Receiving Party is required by law or governmental order to disclose any Confidential
Information, it shall promptly notify the Disclosing Party in writing prior to such disclosure
so that the Disclosing Party may seek a protective order or other remedy."
```

---

### Column 10 — Return and Destruction
Triggers, timing, choice between return/destruction, whether it covers notes/derivatives,
certification requirement, and any archival copy exceptions — all verbatim.

*Example:*
```
"Upon written request of the Disclosing Party or upon termination of this Agreement, the
Receiving Party shall, at the Disclosing Party's option, promptly return or destroy all
Confidential Information and certify in writing that it has done so. This obligation extends to
all copies, notes, and extracts."
```

---

### Column 11 — Assignment and Transfer
Verbatim rules on consent requirements, consent standard, permitted transfers without consent,
consequences of unauthorized transfer, assumption agreements, and continuing liability.

*Example:*
```
"Tenant shall not assign this Lease or sublet the Premises without the prior written consent of
Landlord, which consent shall not be unreasonably withheld, delayed or conditioned. Any
attempted assignment without consent shall be void and of no effect. Tenant shall remain liable
under this Lease following any permitted assignment."
```

---

### Column 12 — Change of Control
Exact definition of "Change of Control" if defined. Otherwise state "No standalone definition".
Report any consent requirement, deemed assignment language, termination rights, or carve-outs
exactly as written.

*Example:*
```
No standalone definition of "Change of Control."
"Any change in ownership or control of Tenant that results in a transfer of more than fifty
percent (50%) of the ownership interests shall be deemed an assignment requiring Landlord's
prior written consent."
```

---

### Column 13 — Termination Rights
Every termination ground, notice period, and cure period verbatim. Note immediate termination
rights, financial consequences, and insolvency triggers exactly as stated.

*Example:*
```
"Either party may terminate this Agreement upon thirty (30) days' written notice if the other
party materially breaches any provision and fails to cure such breach within the thirty (30) day
period. Landlord may terminate immediately upon Tenant's insolvency or filing of bankruptcy."
```

---

### Column 14 — Limitation of Liability
Verbatim waiver of indirect/consequential damages. Verbatim liability cap (if any). List every
carve-out to the waiver or cap exactly as written.

*Example:*
```
"IN NO EVENT SHALL EITHER PARTY BE LIABLE FOR ANY INDIRECT, INCIDENTAL, SPECIAL, CONSEQUENTIAL
OR PUNITIVE DAMAGES, INCLUDING LOST PROFITS."
"Liability of Landlord shall not exceed the total amount of rent paid by Tenant in the twelve
(12) months preceding the claim. This limitation shall not apply to claims arising from gross
negligence or willful misconduct."
```

---

### Column 15 — Indemnification
Who indemnifies whom and for what claims, verbatim. Note third-party vs direct claims, notice and
defense procedure, settlement approval, whether capped, and survival.

*Example:*
```
"Tenant shall indemnify, defend and hold harmless Landlord from and against any third-party
claims arising from Tenant's use of the Premises or negligence. Tenant shall receive prompt
notice of any claim and shall have the right to control the defense, provided that no settlement
shall be made without Landlord's prior written consent."
```

---

### Column 16 — Insurance Requirements
Insurance types and minimum limits required, additional insured status, subrogation waiver,
evidence requirements, and consequences of lapse — all verbatim.

*Example:*
```
"Tenant shall maintain commercial general liability insurance with limits of not less than
$2,000,000 per occurrence. Landlord shall be named as an additional insured. Tenant shall provide
certificates of insurance within ten (10) days of request. If Tenant fails to maintain insurance,
Landlord may obtain it and charge Tenant."
```

---

### Column 17 — IP Ownership and Work Product
Ownership of deliverables, background IP, licenses granted, trademark restrictions, moral rights,
and survival — all verbatim.

*Example:* `Not applicable / No IP ownership provisions.`

---

### Column 18 — Dispute Resolution
Governing law, forum, arbitration details (rules, seat, arbitrators, confidentiality), injunctive
relief rights, cost recovery, and escalation steps — verbatim.

*Example:*
```
"Any dispute arising under this Lease shall be resolved in the state or federal courts located
in San Francisco County, California. Either party may seek injunctive relief in any court of
competent jurisdiction without posting a bond."
```

---

### Column 19 — Survival
Every provision expressly stated to survive termination and its duration. Note general survival
clause (if any) and post-termination obligations verbatim.

*Example:*
```
"The following provisions shall survive termination or expiration of this Lease: Sections 6
(Confidential Information), 14 (Limitation of Liability), 15 (Indemnification), and 5 (Governing
Law). Confidentiality obligations shall survive for five (5) years after termination."
```

---

### Column 20 — Flags and Notes

This is the only column in Sheet 1 where you may go beyond pure extraction. Write a bullet-point
list of factual observations the deal team needs to verify. Permitted content:

- **Drafting anomalies:** Language that appears to contain a typographical error, missing words,
  or internally contradictory terms. Quote the anomalous language and identify the section. Do not
  characterise it as "a drafting error" if the document could also be read as intentional —
  instead state what the text says and flag that it warrants confirmation.
- **Express gaps:** Topics the document is entirely silent on that are normally addressed in
  agreements of this type. State the gap factually — do not characterise what the absence means
  legally.
- **Defined term inconsistencies:** Where a defined term in one section conflicts with its use
  elsewhere, or where a term is used but not defined. State the sections and quote relevant text.
- **M&A-relevant provisions:** Provisions that would require attention in an M&A transaction —
  consent requirements, deemed-assignment language, CoC clauses, surviving obligations, assumption
  requirements, consent fees. State what the document says; do not advise on what to do.

Each flag must be a separate bullet, reference the specific section, and state the factual
observation in one sentence. Do not use advisory language. Do not compare to market standard.
Advisory language and recommendations belong exclusively in Sheet 2 and Sheet 3.

**Column 20 hard limits:** Maximum 5 bullets. One sentence per bullet. No sub-bullets.

recomputed across the full cumulative set each time, not appended batch-by-batch.

## Excel workbook formatting

Build the workbook using `openpyxl`. This section covers Sheet 1 and Sheet 3 formatting, plus the
shared conventions Sheet 2 must also follow. Sheet 2's own sheet-specific formatting (its tab
colour, freeze panes, column widths, and final-row banner) is defined in the child skill,
`mm-coc-tabular-review`, and applied by that skill when it writes Sheet 2.

### Cell colour coding by state (shared across Sheet 1 and Sheet 2)

| State | Fill colour |
|-------|------------|
| Answered | White |
| Not present | Gray |
| Unclear | Yellow |
| Needs review | Yellow |
| Source-citation rows | Light blue |

### Headers
Dark navy background, white bold text. Header row taller than standard to accommodate two-line
labels. Applies to Sheet 1, Sheet 2, and Sheet 3 headers alike.

### Work-product banner
Row 1 of Sheet 1 (and Sheet 2, when produced) spans the full width of the sheet and contains: the
review title, the list of documents reviewed, the date, and the verification reminder. Amber
background with dark red bold text.

### Distribution notice
Row 2 of Sheet 1 (and Sheet 2, when produced) spans the full width and reads:
> "This review is derived from source documents that may be privileged, confidential, or both. It
> inherits the sources' privilege and confidentiality status — distribution beyond the privilege
> circle can waive privilege. Store with the matter's privileged files and make distribution
> decisions deliberately."

White background, small gray italic text.

### Row heights
- Data rows: approximately 100 points
- Citation rows: approximately 30 points

### Column widths (Sheet 1)
- Identifier columns (Document, Doc Type, Parties): ~28–36 characters
- Analysis columns: ~40–50 characters
- Flags/Notes column and any custom column the user described as needing more room: widest on the
  sheet

### Freeze panes (Sheet 1)
Freeze after the first three identifier columns and below the first three header rows, so analysts
can scroll right while keeping the document name visible. If the user replaced the defaults
entirely (Phase 0, option 3) and there are fewer than three natural identifier columns, freeze
after whichever leading columns serve as document identifiers.

### Tab colours
- Sheet 1 tab: Navy
- Sheet 3 tab: Dark green
- Sheet 2 tab (only when produced): Dark red — set by the child skill

### Output path
Save the workbook to the same path for every batch, overwriting the previous version of the file
with the newly appended data (not the previous rows — those are preserved inside the workbook and
carried forward each time it is re-saved):
```
/mnt/user-data/outputs/<matter_name>_MA_Contract_Review.xlsx
```
After each batch is added, call `present_files` with the saved path so the user always has the
latest cumulative workbook. This is the same path the child skill writes Sheet 2 into — never a
separate file.

## What this skill does not do

- **Does not replace legal review.** Every cell is a lead that points a lawyer to where they need
  to look, not a conclusion they can act on without reading the source.
- **Does not produce legal advice or a legal opinion.**
- **Does not silently skip a document.** Every uploaded file gets a row. If it cannot be read,
  the row says so and every cell is marked `needs_review`.
- **Does not invent custom key terms.** If the user chooses to add or replace the default key
  terms and gives an incomplete description, this skill asks a follow-up rather than guessing what
  a column should capture.
- **Does not run the Change of Control analysis without explicit confirmation.** If the user
  declines at the CoC gate, the child skill is never invoked, Sheet 2 is not produced, and no CoC
  risk ratings appear anywhere.
- **Does not build Sheet 2 itself.** All Sheet 2 field definitions, extraction discipline, and
  formatting live in the child skill, `mm-coc-tabular-review`. This skill only decides whether to
  invoke it and consumes its output for Sheet 3.
- **Does not produce Google Sheets output.** The deliverable is always an Excel `.xlsx` file.
- **Does not review a base agreement in isolation when amendments or schedules are present.** The
  integrated, final position is what matters.
- **Does not include a state-count table in the Summary sheet under any circumstances.**
- **Does not process more than 5 documents/packages per batch, and does not create a separate
  workbook per batch.** All uploaded material is worked through in batches of up to 5, appended to
  one consolidated workbook, unless the user has asked to review each batch before continuing.
